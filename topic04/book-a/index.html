Objectives

  • Complete the XML file for the CouchManager blog.
  • Create a DTD file for the CouchManager XML file.

Continuous Assessment

Open your Lab4 project in eclipse that contains your couch manager blog xml file.

The following shows an example of the elements you may decide to have in your couchmanager xml file. Make sure you have all of these elements or the equivalent ones that you have named. Also ensure you have some data in each element. Take the data from the blog website (cut and past text, and save images).

DTD file

Your blog file should be something like:

  • blog - title, author, post
  • post - post_title, date, content, comment
  • date - day, month, year
  • content - paragraph
  • paragraph - text, image
  • comment - date, author, content

Think about the order of the items, it may not always be the case that the xml file has the order of post_title, date, content and comment correct. How can you write your definition to allow for any order of a list of elements?

Create a new dtd file in eclipse. Name it couch_manager.dtd

In your XML file add the correct doctype link to the dtd file


 <!DOCTYPE root_name SYSTEM "path to dtd file">

Implement your rules

  • *indicates zero or more occurrences for an element
  • + indicates one or more occurrences for an element
  • ? indicates either zero occurrences or exactly one occurrence for an element.

An element name with no qualifying punctuation must occur exactly once. Commas between element names indicate they must occur in succession; if commas omitted, elements can occur in any order.

Save your DTD and check to make sure your XML file validates against your DTD. To do this you simply open the XML file in a browser.

Exercises

Remember to:

  • Comment your DTD file
  • Section your DTD according to functionality
  • Declare all elements contained in your XML file
  • Declare all attributes contained in your XML file

Examples


<census-record taker="3163">
  <date><year>2003</year><month>10</month><day>11</day></date>
  <address>
    <street>471 Skipstone Lane <unit>4-A</unit></street>
    <city>Emerald City</city>
    <county>Greenhill</county>
    <country>Oz</country>
    <postalcode>885JKL</postalcode>
  </address>
  .....

The following rules would apply to the data stored in the above XML file.


<!ELEMENT date (year, month, day)>
<!ELEMENT year (#PCDATA)>
<!ELEMENT month (#PCDATA)>
<!ELEMENT day (#PCDATA)>
<!ELEMENT address (street, city, county, country, postalcode)>
<!ELEMENT street (#PCDATA | unit)*>
<!ELEMENT city (#PCDATA)>
<!ELEMENT county (#PCDATA)>
<!ELEMENT country(#PCDATA)>
<!ELEMENT postalcode #PCDATA>
<!ELEMENT unit (#PCDATA)>

The street rule allows for mixed content in the street element. The declaration must have #PCDATA first followed by all sub-elements, in this case unit. Each sub-element is separated by | a vertical bar. (This signifies an "OR" operator). The asterisk * here is required. It means that there can be zero or more of whatever comes before it. The character data and the unit data is all optional. The census taker could leave the street element blank and the DTD would validate the file. Changing the * to a + is not allowed because this is a mixed content element, street contains character data and other elements, you cannot force the mixed content element to contain character data.


<person employed="fulltime" pid="P270405">
    <name>
      <first>Meeble</first>
      <last>Bigbug</last>
      <junior/>
    </name>
    <age>39</age>
    <gender>male</gender>
  </person>

To write the rules for person element:


<!ELEMENT person (name, age, gender)>
<!ELEMENT name (first, last, (junior | senior)?)>
<!ELEMENT age (#PCDATA)>
<!ELEMENT gender (#PCDATA)>
<!ELEMENT first(#PCDATA)>
<!ELEMENT last(#PCDATA)>
<!ELEMENT junior EMPTY>
<!ELEMENT senior EMPTY>
<!ATTLIST person
    pid       ID                   #REQUIRED
    employed  (fulltime|parttime)  #IMPLIED>

The name contains a first and last name and optionally it can have a zero or one junior or senior element. The id attribute for person is required as it is defined of type ID. The employed attribute is optional and can contain either fulltime or parttime.
